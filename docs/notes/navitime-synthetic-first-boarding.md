# NAVITIME route_transit は経路の先頭乗車を「合成時刻」で返す（実ダイヤでない）

## 結論

NAVITIME RapidAPI（`navitime-route-totalnavi.p.rapidapi.com/route_transit`）は、**経路の最初の
乗車（先頭電車）の発車時刻を実時刻表ではなく「ホーム到着 + 固定の待ち（実測で約3分）」で合成して
返す**。アプリの発着時刻表示（#115 / `_boardingNode` → `formatClock`）はこの値をそのまま出して
いるだけで、プロキシ（`functions/src/index.ts` の `navitimeProxy`）も無加工で透過している。

したがって「10:33発の電車が実在しない」という事象の根本原因は **データソース側**にあり、
アプリの表示ロジックやプロキシのロジックの修正では解消できない。

## 検証（実機 / 2026-06-20, iPhone17 simulator, 本番プロキシ + App Check）

### 生レスポンスの区間分解（大森→大井町, start_time=10:30）

```
point start
move/walk  徒歩            10:30→10:32  (2分, 104m)   ← アクセス徒歩
point 大森（東京都）                                   ← 10:32 ホーム到着
move/local_train 京浜東北線快速 10:35→10:38 (3分)      ← 10:35発 = 到着10:32 + 3分
point 大井町
move/walk  徒歩            10:38→10:41
point goal
```

- `options=railway_calling_at` を付けても **calling_at（実停車時刻）は空**。応答に実発車時刻は無い。

### start_time を1分刻みにした追従テスト（大森→大井町）

| 照会 start_time | ホーム到着(=start+2分) | API の発車 | 待ち |
|---|---|---|---|
| 10:25 | 10:27 | 10:30 | 3分 |
| 10:26 | 10:28 | 10:31 | 3分 |
| 10:27 | 10:29 | 10:32 | 3分 |
| 10:28 | 10:30 | **10:33** | 3分 |
| 10:30 | 10:32 | 10:35 | 3分 |
| 10:33 | 10:35 | 10:38 | 3分 |

返ってきた「発車ボード」は 10:30,10:31,10:32,…,10:45 と**毎分**。

**判定:** 実在の京浜東北線（昼は約5分間隔）なら、到着時刻に応じて待ちが可変になり、発車は離散した
次発に張り付く（プラトーが出る）はず。ところが待ちは何分に着いても**一律3分**で、発車が照会時刻に
1:1で追従する。毎分発車する京浜東北線は実在しないので、先頭乗車は**合成**と断定できる。
「10:33発」が出るのは start=10:28 で照会したから（10:28+5）であり、実在ダイヤに10:33があるからではない。

## 未解決 / 未検証

- **実時刻表を引く手段の有無:** 別エンドポイント（時刻表）・別パラメータ・別データソースで先頭便を
  実ダイヤにスナップできるかは未調査。検証は下記クォータ回復後に、間隔を空け少数で行うこと。

- **【重要・本番影響】NAVITIME RapidAPI の月間クォータを使い切った。** 502 の本文は
  `{"message":"You have exceeded the MONTHLY quota for Requests on your current plan, BASIC. Upgrade ..."}`。
  プロキシ（`functions/src/index.ts` の navitimeProxy）は RapidAPI のこの 502 を無加工で返す。
  ＝**ライブアプリの乗換ルートも現在 502 で動かない**。回復は月次リセット（請求サイクル先頭）まで、
  または RapidAPI の BASIC プラン → 上位へアップグレードが必要。本調査の連続照会が残量を消費した一因。
  プローブは 1 plan() ≒ NAVITIME 10数コール（徒歩最大化/再照会で多発）に注意。
- **乗換区間は実ダイヤの可能性:** 同一経路の2本目以降（例: りんかい線）は待ち10分など可変で、実ダイヤ
  寄りに見えた。合成は「経路の最初の乗車」限定の可能性があるが、乗換側の追従検証は未実施。

## 教訓（検証手順）

- 1サンプル（例 10:30→10:35）だけでは「実在の次発」と「start+固定の合成」を区別できない。
  **必ず start_time を刻んで追従（プラトーの有無）を確認する。**
- 幽霊便を `from_time == start_time` の完全一致で判定すると、+固定オフセット型の合成を取りこぼす
  （偽陰性）。判定は「複数照会で from_time が start_time に線形追従するか」で行う。

関連: #115（発着時刻表示）, #121（深夜の start_time 追従幽霊便。本件は昼間でも起きる点が新規）。
