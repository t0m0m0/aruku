# 徒歩最大化の再設計：乗車駅探索（board-station search）

- **ステータス:** 設計（実装前・計測待ち）。本書は新方式の指針。確定したら [route-optimization.md](../spec/route-optimization.md) §1/§3 と §4 #115 を更新する。
- **背景の前段:** [refetch-reanchor-design.md](refetch-reanchor-design.md) §7 で「基準経路の再アンカーでは徒歩最大化が構造的に達成できない」ことが実機で確定した。本書はその §7 (A)/(B) を受けた**候補生成・データ源の再設計**。
- **関連不変条件:** #65 / #67 / #71 / #115 / #117 / #118 / #121 / 逆戻りフィルタ（§4 を回帰させない）。

## 1. 目的の再定義（§1 をそのまま書き下す）

「制限時間（予算）内で最大限歩く」。全徒歩が予算超過のとき、**徒歩で X 駅まで行き（t1）、X→goal を電車で行く（t2）。t1 + t2 ≤ 予算 を満たし、徒歩を最大化する X を選ぶ。**

- 例: 蒲田→上野・予算3時間。全徒歩 4h25m（1h25m 超過）。蒲田→X 徒歩 t1 ＋ X→上野 電車 t2 が3時間に収まる最遠の X を採る。
- **厳密には「t1 最大」ではなく「総徒歩 `walkMinutes` 最大」（§1）。** t1（前半徒歩）は支配的レバーだが、NAVITIME が返す t2 内の徒歩（アクセス／降車後）も歩数に入る。降車側の徒歩（現状ハイブリッドの後半徒歩）も依然有効な歩数源なので捨てない。

## 2. 方式：候補乗車駅 X ごとに route_transit を引き直す

§7 の根本問題は「**基準経路（最速1本）を固定して乗車駅を後ろへ再アンカー**」した点。route_transit は座標から常に最速＝前方の駅へ歩かせる経路を返すため、再アンカーは NAVITIME の最適化と衝突して詰む（大森→大井町の例）。

新方式は**乗車駅 X を探索変数にし、X の座標から都度 `route_transit(X → goal, 出発 = departureAt + t1)` を引き直す**：

- 返ってくるのは**その時刻に X 付近から実際に乗れる自己整合な door-to-door 経路**。`firstMissedTrain`（乗り遅れ）という概念自体が消える。§7 の詰みは起きない。
- 到着 = `departureAt + t1 + t2` をそのまま予算判定に使える。**現状の `_maxEnrichAttempts=8` 再検証ループは不要**になる（採用パスの enrich は徒歩ジオメトリ取得のみに縮小）。
- NAVITIME が X でなく前方駅 Y から乗る経路を返しても OK。その場合の前半徒歩は `origin→Y`（NAVITIME が実際に乗った駅）で測り直す＝ t1 の実体。破綻しない。

### 探索（単調性の利用）

- t1 は X が goal 寄りになるほど単調増。X を origin から外向きに伸ばし、**「予算に収まる最遠の X」を二分＋近傍数点で探索**する（ダイヤのラムプ性で完全単調ではないため近傍を確認）。
- 全候補駅を引かずに NAVITIME 往復を最小化するのが狙い。実呼数は §4 で実測する。

## 3. 候補乗車駅 X の母集合：stops ＋ 近接駅探索

1. **初回 `route_transit(origin→goal)` の stops**（#67 で座標を捨てない前提）をコリドー上の母集合にする。
2. **近接駅を Places nearby で追加**：`placesProxy`（§2.1・既存）で `transit_station`/`subway_station` を引き、origin 近傍やコリドー上の**別路線駅（山手線等）**を候補に加える。**新規 API 依存なし**で賄える見込み。
   - 別路線許容は [refetch-reanchor-design.md](refetch-reanchor-design.md) の B案と同趣旨だが、再照会の制約緩和ではなく**候補生成側で別路線駅を母集合に入れる**形で実現する。

## 4. 実測プラン（スコープ確定のための計測）

スコープは「探索コストを実測してから決める」。蒲田→上野・3時間を実ケースに、**使い捨てプローブ**（実機ログ採取・本体ロジックは触らない）で次を測る。

| 計測項目 | 取り方 | スコープ判断に効く点 |
|---|---|---|
| route_transit 1往復の実レイテンシ | `_fetchTransitAt(X→goal)` を実機/プロキシ経由で計測（p50/p90） | N 件逐次の総 wall-clock の基礎 |
| 候補駅数 N | 初回 stops 数 ＋ Places nearby で増える駅数 | 30/min に収まる N か／間引きの要否 |
| 逐次 vs 少数並列の所要 | N 件を逐次／少数並列で実行し総時間 | UX 許容（数秒以内か）と 30/min の両立点 |
| 単調探索での実呼数 | origin から外向き二分＋近傍数点での実 NAVITIME 回数 | 全 N 引かずに済む削減率 |

### 判断の閾値（スコープ分岐）

- **逐次総時間が数秒以内かつ実呼数が 30/min 内** → 「前半＋後半の両側探索（乗車駅 X・降車駅 A を両方）」まで広げてよい。
- **レイテンシが重い／N が大きい** → 「前半徒歩のみ・単調二分で最小呼数」に絞る。

### 4.2 結論

1. **方式が成立する。** §7 の構造的限界（再アンカーの詰み）は回避できた。各 X から引き直した経路は自己整合で乗り遅れが起きず、**予算内で総徒歩 156分（新橋乗車）** に届く。現状本番の縮退（徒歩3分・余裕147分）に対し圧倒的改善。
2. **単調性がクリーン。** t1 は単調増、t2 は単調減、total も単調増で、予算内フラグは1回だけ反転（有楽町）。→ **二分探索が妥当**（全 N 不要）。
3. **コストは軽い。** p50≈0.5s/コール。逐次9コールで約4.8s、二分なら≈6コールで約3s。30/min にも収まる。UX 上も許容圏。
4. **前半徒歩のみで実質最大に届く。** ゴール（上野公園・非駅）への最終徒歩は NAVITIME の door-to-door（t2）に含まれるため、降車側を別途探索しなくても総徒歩はほぼ最大化される。→ **両側探索は MVP では不要**。
5. **近接駅／別路線探索は MVP 不要。** 京浜東北線の停車駅だけでコリドー全域をカバーでき、最適点（新橋）も拾えた。Places nearby の新規バックエンドは見送り。

#### 重要な注意：t1 は直線推定なので境界は手前へ寄る

`done` までの t1 は **haversine÷80 の下限**。実街路徒歩（Google）は迂回ぶん長くなるため、**実 walk で測り直すと境界（新橋）は手前へ移動**し得る（新橋 直線152分→実測は180超で予算外の可能性）。よって本実装では：

- **探索（粗い絞り込み）は直線 t1 で単調二分**してコールを節約し、
- **採用候補の最終判定は実 walk（Google）で t1 を測り直し**、その実 `boardAt` で `route_transit(X→goal)` を引き直して予算内を確定する。

直線で予算外なら確実に予算外（下限ゆえ）なので二分の上界打ち切りは安全。境界付近の数駅だけ実 walk で確認すればよい。

### 4.3 実機ログからの発火率集計（#309→#310）

collapse/board-search の発火率は `RouteSearchMetrics.toLogLine()`（#309）が `[route-metrics]` 行で端末ログに出す（サーバへは送らない・#268 PII 方針）。実機ログ（`flutter run` / logcat の保存）を **`tool/route_metrics_agg.dart`** に流すと、発火率・上流往復本数・所要の分布と、**collapse=1 サブセットの walkCalls**（＝#310 でマトリクスに束ねて削減できる往復の的）を集計できる。

```
dart run tool/route_metrics_agg.dart device.log        # ファイル
flutter run | dart run tool/route_metrics_agg.dart     # パイプ
```

採取から集計までは **`tool/run_metrics.sh`** が実機で一括する（profile 起動＝定性ログ抑制・実機相当レイテンシ）。無線接続で Dart VM Service が繋がらず `flutter run` が止まる場合は、USB 接続にするか `tool/run_metrics.sh --logs-only`（`flutter logs` の syslog ストリームで VM Service 不要・無線可）を使う。

`toLogLine()` は **フェーズ別所要**（`guidanceMs`/`hybridMs`/`enrichMs`/`boardSearchMs`/`alternativesMs`/`finalizeMs`）を出し、集計器が total 比の降順で「どの段が時間を食うか」を示す。パイプラインは guidance→hybrid→enrich→(boardSearch)→alternatives→finalize の直列なので、各段の和＋残差＝total で未帰属分も可視化される。時間短縮はまずこの内訳で支配フェーズを特定してから狙う。

#310（乗車駅探索の徒歩評価を matrix へバッチ化）は発火率が低ければ費用対効果が限定的なため、この集計で発火率を確認してから着手可否を決める。

### 6.4 実機検証で判明したステップバックの必要性（2026-06-19・確定）

崩壊判定は実機でも正しく発火した（蒲田→上野公園・徒歩6分・余裕136分→`collapse=true`）。だが当初の「直線二分探索の境界駅を1本だけ採用」では**徒歩6分のまま逆戻り**した。`[BOARD-FB]` ログで機序を確定：

- 直線推定では **新橋（徒歩158分・到着172分）が境界**で予算内に見える。
- だが採用候補の enrich で**実街路 walk が新橋189分**→到着207分で予算超過→除外→候補が1本だけのため標準（徒歩6分）へ逆戻り。

**修正＝境界の実 walk 確定（ステップバック）**：直線二分で概略境界を出した後、境界から手前へ実 walk＋実 boardAt で引き直し、予算内かつ乗り遅れ無しの最遠駅を確定する（最大 `_boardSearchVerifySteps`=4 駅）。

実機ログ（修正後）：
```
verify #6 新橋  realWalk=189 arr=207  （予算180超）→棄却
verify #5 浜松町 realWalk=173 arr=190  （予算180超）→棄却
verify #4 田町  realWalk=150 arr=169  （予算内）   →確定
afterReselect walk=154 arrival=169
```

→ **徒歩6分 → 徒歩154分・余裕11分**。直線は下限なので境界が実 walk で手前へ寄る（§4.2 で予見）を実装で吸収。回帰テスト（直線境界が実 walk で乗り遅れ→手前へ後退）を追加済み。

### 6.5 残課題（調整・拡張）

- **定数の感度調整**：`_collapseWalkMarginMin`(10)/`_collapseSlackRatio`(0.4)/`_boardSearchVerifySteps`(4) は実機の他ケースで要再確認。
- **探索の往復コスト**：崩壊時のみだが二分探索（route_transit 数回）＋ステップバック（実 walk＋route_transit 数回）。レート制限 30/min・UX レイテンシは蒲田で問題なし（p50≈0.5s）だが、停車駅が非常に多い経路で要確認。
- **降車側徒歩・近接駅探索**は未着手（§4.2 で MVP 非対象）。前半徒歩のみで不足する実ケースが出たら検討。
- **診断ログ撤去済み**（`[BOARD-FB]` は調査用。確定後に削除）。

### 6.1 組み込み試作で判明した制約（2026-06-19・ステップ2の設計確定材料）

`_buildBoardSearchCandidate` を「全徒歩が予算超過なら走らせ候補に1本足す」常時併存で試作したところ、既存テストが4件落ち、**常時併存は不可**と判明した（試作はコミットせず revert）。

- **measure-first の往復契約と衝突（§3.1/§3.3）:** 「選定中は API を引かず、採用1経路だけ実測する」契約に対し、board-search は選定中に駅ごとの `route_transit` を引く。前半 t1 を直線推定にして Google 呼び出しは消せたが（探索中は街路実測しない）、**NAVITIME の駅別引き直しは機構上不可避**。`navitimeProxy 1回` を前提にした #115 テストが落ちる。
- **既存ハイブリッド機構との重複・上書き:** ハイブリッド生成（フロンティア＋マトリクス）も同じ停車駅で徒歩最大化しており、board-search はそれと競合して上書きする。`_mock`（駅別引き直しを模さず origin 起点経路を返す）下では board-search が**偽候補を作って勝つ**ため、#67/#71/マトリクス選定テストが落ちる。

**結論（ステップ2の方針）:** board-search は**常時併存ではなく、measure-first が collapse したときだけ走る限定フォールバック**にする。引き金は「予算内の採用候補が**ごく僅かな徒歩**しか持たない（標準全電車へ縮退）のに予算が大きく余っている」状態（§7 で観測した症状そのもの。例: 徒歩3分・余裕137分）。

- 既存パイプラインの採用候補の `walkMinutes` が小さい（collapse 検出）ときだけ board-search を起動 → 既存ハイブリッドテスト（採用 walk が十分大きい）は不発で緑のまま。蒲田→上野公園（walk≈3分）でのみ起動。
- テストは `_mock` ではなく **`_requeryMock`（start 座標で経路を出し分け）** で駅別引き直しを正しく模す。
- §3.1/§3.3 に「collapse 時の徒歩最大フォールバック＝乗車駅探索」を追記。#115 系テストの `navitimeProxy 回数`前提はフォールバック起動時のみ緩める。

### 6.2 崩壊（collapse）の判定定義（確定）

採用前の見積もり選定（`selectBestRoute`）の結果に対し、次を満たすとき board-search を起動する：

1. 採用見積もり候補が**予算内**（`arrivalMinutes ≤ budget`）であり、かつ
2. その `walkMinutes` が、**予算内の標準乗換候補（`parsed` 由来）の最大 `walkMinutes`** を `_collapseWalkMarginMin`（まず 10 分）**以下しか上回らない**。
   - = ハイブリッド機構が標準乗換（ほぼ全電車）を実質改善できていない＝徒歩最大化が崩壊。
3. 基準経路 `base != null`（探索する停車駅がある）。

**なぜ既存テストを壊さないか:** 蒲田はハイブリッド全滅で採用＝標準（差 0）→ 起動。既存の正常系（例: ハイブリッド徒歩40分が標準25分を15分上回る、または測定アクセス徒歩30分が標準10分を20分上回る）は差がマージン超 → **不発**。したがって `navitimeProxy 回数`等の既存前提に触れない。

**起動後:** board-search 候補（直線 t1 で構築）を `candidates` に足し、既存の選定＋enrich ループに通す。採用されれば enrich が実街路で t1 を測り直し、超過なら除外して選び直す（既存の安全網を流用）。定数 `_collapseWalkMarginMin` で感度を微調整。
