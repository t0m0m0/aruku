# 徒歩最大化の再設計：乗車駅探索（board-station search）

- **ステータス:** 設計（実装前・計測待ち）。本書は新方式の指針。確定したら [route-optimization.md](../spec/route-optimization.md) §1/§3 と §4 #115 を更新する。
- **背景の前段:** [refetch-reanchor-design.md](refetch-reanchor-design.md) §7 で「基準経路の再アンカーでは徒歩最大化が構造的に達成できない」ことが実機で確定した。本書はその §7 (A)/(B) を受けた**候補生成・データ源の再設計**。
- **関連不変条件:** #65 / #67 / #71 / #115 / #117 / #118 / #121 / 逆戻りフィルタ（§4 を回帰させない）。

---

## 1. 目的の再定義（§1 をそのまま書き下す）

「制限時間（予算）内で最大限歩く」。全徒歩が予算超過のとき、**徒歩で X 駅まで行き（t1）、X→goal を電車で行く（t2）。t1 + t2 ≤ 予算 を満たし、徒歩を最大化する X を選ぶ。**

- 例: 蒲田→上野・予算3時間。全徒歩 4h25m（1h25m 超過）。蒲田→X 徒歩 t1 ＋ X→上野 電車 t2 が3時間に収まる最遠の X を採る。
- **厳密には「t1 最大」ではなく「総徒歩 `walkMinutes` 最大」（§1）。** t1（前半徒歩）は支配的レバーだが、NAVITIME が返す t2 内の徒歩（アクセス／降車後）も歩数に入る。降車側の徒歩（現状ハイブリッドの後半徒歩）も依然有効な歩数源なので捨てない。

---

## 2. 方式：候補乗車駅 X ごとに route_transit を引き直す

§7 の根本問題は「**基準経路（最速1本）を固定して乗車駅を後ろへ再アンカー**」した点。route_transit は座標から常に最速＝前方の駅へ歩かせる経路を返すため、再アンカーは NAVITIME の最適化と衝突して詰む（大森→大井町の例）。

新方式は**乗車駅 X を探索変数にし、X の座標から都度 `route_transit(X → goal, 出発 = departureAt + t1)` を引き直す**：

- 返ってくるのは**その時刻に X 付近から実際に乗れる自己整合な door-to-door 経路**。`firstMissedTrain`（乗り遅れ）という概念自体が消える。§7 の詰みは起きない。
- 到着 = `departureAt + t1 + t2` をそのまま予算判定に使える。**現状の `_maxEnrichAttempts=8` 再検証ループは不要**になる（採用パスの enrich は徒歩ジオメトリ取得のみに縮小）。
- NAVITIME が X でなく前方駅 Y から乗る経路を返しても OK。その場合の前半徒歩は `origin→Y`（NAVITIME が実際に乗った駅）で測り直す＝ t1 の実体。破綻しない。

### 探索（単調性の利用）

- t1 は X が goal 寄りになるほど単調増。X を origin から外向きに伸ばし、**「予算に収まる最遠の X」を二分＋近傍数点で探索**する（ダイヤのラムプ性で完全単調ではないため近傍を確認）。
- 全候補駅を引かずに NAVITIME 往復を最小化するのが狙い。実呼数は §4 で実測する。

## 3. 候補乗車駅 X の母集合：stops ＋ 近接駅探索

1. **初回 `route_transit(origin→goal)` の stops**（#67 で座標を捨てない前提）をコリドー上の母集合にする。
2. **近接駅を Places nearby で追加**：`placesProxy`（§2.1・既存）で `transit_station`/`subway_station` を引き、origin 近傍やコリドー上の**別路線駅（山手線等）**を候補に加える。**新規 API 依存なし**で賄える見込み。
   - 別路線許容は [refetch-reanchor-design.md](refetch-reanchor-design.md) の B案と同趣旨だが、再照会の制約緩和ではなく**候補生成側で別路線駅を母集合に入れる**形で実現する。

## 4. 実測プラン（スコープ確定のための計測）

スコープは「探索コストを実測してから決める」。蒲田→上野・3時間を実ケースに、**使い捨てプローブ**（実機ログ採取・本体ロジックは触らない）で次を測る。

| 計測項目 | 取り方 | スコープ判断に効く点 |
|---|---|---|
| route_transit 1往復の実レイテンシ | `_fetchTransitAt(X→goal)` を実機/プロキシ経由で計測（p50/p90） | N 件逐次の総 wall-clock の基礎 |
| 候補駅数 N | 初回 stops 数 ＋ Places nearby で増える駅数 | 30/min に収まる N か／間引きの要否 |
| 逐次 vs 少数並列の所要 | N 件を逐次／少数並列で実行し総時間 | UX 許容（数秒以内か）と 30/min の両立点 |
| 単調探索での実呼数 | origin から外向き二分＋近傍数点での実 NAVITIME 回数 | 全 N 引かずに済む削減率 |

### 判断の閾値（スコープ分岐）

- **逐次総時間が数秒以内かつ実呼数が 30/min 内** → 「前半＋後半の両側探索（乗車駅 X・降車駅 A を両方）」まで広げてよい。
- **レイテンシが重い／N が大きい** → 「前半徒歩のみ・単調二分で最小呼数」に絞る。

## 5. 非目標（この再設計でやらないこと）

- レート制限プロキシ（30/90 req/min）の変更。N が超えるなら**候補側の間引き・探索削減で吸収**する。
- §4 不変条件（#65/#67/#71/#121/逆戻り）の挙動変更。新候補も `selectBestRoute` の逆戻り・予算判定を通す。
- 地図/徒歩ジオメトリのデータ源変更（徒歩は引き続き Google）。
