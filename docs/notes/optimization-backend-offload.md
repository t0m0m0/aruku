# メモ: 最適化ロジックのバックエンド移管について（検討）

- **ステータス:** 検討メモ（未決定・着手前）
- **日付:** 2026-06-15
- **関連:** ADR-001（ルート最適化アーキテクチャ）, `lib/core/services/navitime_route_service.dart`, `functions/src/index.ts`

「アプリのパフォーマンスを考えると重いロジックはバックエンドに寄せたほうがいいのでは」という
議論を受けた、現状整理と将来の選択肢のメモ。決定ではない。

## 現状の役割分担

- **バックエンド（Firebase Functions）** は素通しのプロキシ。計算ロジックは持たない。
  - 担うのは横断的関心事のみ: APIキー秘匿（Secret Manager）・App Check 認証・IP レート制限・
    マトリクス要素数上限（課金暴発防止）。
  - レスポンスは無加工で返す（変換・選定はクライアント側）。
- **フロントエンド（端末側 Dart）** が最適化の全工程を実行。
  - ハイブリッド候補生成（`_buildHybrids`）・「予算内で徒歩最大」選定（`selectBestRoute`）・
    楽観推定→実測フォールバック・側別迂回率学習（`_learnDetours`/`_inflateWalk`）・
    境界帯マトリクス（`_measureFrontierBand`）・時刻表整合（`arrivalMinutes`/`firstMissedTrain`）。

## 論点: 何が「重い」のか

- **CPU 負荷は軽い。** 候補生成は O(駅数²) だが `_maxHybridCandidates=40` でキャップされ、
  最大でも数百候補・各候補は数区間の `fold`。端末で数ミリ秒。ここを移管しても体感は変わらない。
- **本当のボトルネックは外部APIへの逐次往復。** `_finalize` のループ（最大 `_maxEnrichAttempts=8`）が
  「選定（端末）↔ 実測（Google）」を繰り返し、端末視点では毎回
  `端末 → Firebase → Google → Firebase → 端末` の往復になる。境界帯マトリクス・乗り遅れ再照会も加わる。

## バックエンド移管の利点と代償

- **利点（往復の削減）:** ループ全体を Function 内（`asia-northeast1`）で回せば各プローブが
  `Function → Google` だけになり 1 往復あたりのレイテンシが下がる。最大 8 回ぶんの
  `端末↔Function` ホップを畳める。CPU ではなくこの往復削減が唯一の本物のメリット。
- **代償:**
  - Functions の実行時間課金が増える（薄いプロキシ→全API往復が終わるまで滞在）。コールド スタートの影響も増す。
  - テスタビリティ低下。現状は純粋関数（`selectBestRoute`/`buildRoutePlan`）として Dart 側で完結し
    手厚くテスト済み。TS へ移すと二重実装かテスト資産の作り直しが必要。

## 現実的な落としどころ

- 全面移管より、**ループだけをバックエンドに持つオーケストレーション エンドポイント**を 1 個足す案が有力。
  候補生成・最終選定は端末のまま、`実測 ↔ 選び直し` の反復だけ Function 内で回し、往復を 8 回→1 回に畳む。
- ただし着手の前に **実測のレイテンシが実際に問題かを計測**する。Google コール削減（キャッシュ #116・
  側別学習 #117・帯マトリクス #118）は作り込み済みで、多くの検索は数回の往復で収束する想定。
  「検索開始→結果表示」の実測値と、そのうちネットワーク待ちの占有率を取ってから判断する。
- ADR-001 の `RouteService` 抽象により方式変更は後から差し替え可能。今は決定不要。

## ネクストアクション（必要になったら）

1. フェーズ別（routing / walkability / building）の所要時間と Google/NAVITIME コール回数を実機計測。
2. ネットワーク待ちが体感を支配しているなら、オーケストレーション エンドポイントの PoC。
3. 実行時間課金の見積もりと、薄いプロキシ維持との比較。

# 徒歩最大化ロジックの既知の限界と再設計案（検討）

- **ステータス:** 検討メモ（未決定）
- **日付:** 2026-06-15
- **対象:** `lib/core/services/navitime_route_service.dart`, `lib/core/services/hybrid_route_selector.dart`

現行ロジックを読み直して洗い出した穴と、ゼロから設計し直すならの案。決定ではない。

## 既知の限界（影響が大きい順）

1. **目的関数が ADR とコードで不一致（最重要・要判断）。**
   ADR-001 は「`walkKm` が最大となるものを選択」と記すが、`selectBestRoute` が最大化するのは
   `walkMinutes`（徒歩時間）。kcal は km から算出（`kcalPerKm`）。運動量最大化が主眼のはずが選定は
   時間を最適化しており、`_inflateWalk` が徒歩「分」を迂回率で割増すためズレが増幅する。
   「実距離は短いがクネクネして時間がかかる経路」が「直線的に長く歩く経路」より優先されうる。
   徒歩最大化の定義そのものが曖昧。**km か min かを正として一貫させる判断が要る。**

2. **ハイブリッド候補が単一の最速経路にアンカーされている。**（**#292 で大幅緩和**）
   `_baseForHybrid` は停車駅2つ以上の最短経路を1本だけ選び、その路線の同一セクション内の乗降ペアのみ
   候補化する。最速でないが歩ける回廊に沿った別路線で乗り降りして歩く候補が原理的に出ない
   （別路線は標準乗換候補としては残るが、その上で徒歩を増やすバリエーションは作られない）。乗換を
   またぐ徒歩挿入も生成されない。徒歩を増やす探索空間が最速 itinerary の幾何に縛られている。

   → **#292 で `_basesForHybrid` を追加**し、取得済み options を routeName 集合でフィンガープリント
   して路線ファミリ別の代表を最大 `_maxHybridBases`（=3）本まで土台にするよう拡張した。別路線コリドー
   由来の「予算内・徒歩多め」候補がプールへ入り選定対象になる。増分 API コストはゼロ（新規
   `/guidance/plan` 照会を増やさない・#288 §4「素材は1回の照会に既に入っている」）。**残る限界**:
   ファミリ内の代表選定は現行の総所要 `minutes` 最小のまま（限界1＝目的関数 km/min 未決に依存）で、
   `allowModes` サブネット限定を追加コリドー源にする案（別 issue）は未着手。乗換をまたぐ徒歩挿入も
   引き続き生成しない。

3. **「楽観推定 → 1段ずつ剥がす」収束が 8 回上限で打ち切られる。**
   最大 40 駅×組合せで数百候補が出るのに、実測試行は `_maxEnrichAttempts=8` + マトリクス1回（片側
   10 駅上限）。フロンティア付近に「推定内・実測超過」候補が密集すると真の徒歩最大に到達する前に
   上限へ達し、`shortest`（最早到着）へフォールバック＝徒歩最大化を諦めた速い経路を返す。目的を
   裏切る方向に縮退する品質バグに近い。

4. **迂回率学習が単調増加・クランプ [1.0, 2.0] で過大に倒れる。**
   病的な 1 レッグで早期に高い迂回率を学習すると以降の徒歩推定が一律割増され、本来予算内の徒歩最大
   候補が探索順で後回しになる。8 回上限と相まって安全側（歩かない側）に収束しやすいバイアス。

5. **直線距離前提ゆえバリア（川・線路・高速）で空振りしやすい。**
   選定は実測前は haversine 直線。歩道のない川・線路で実際は大きく迂回する候補が直線では魅力的に
   見えて選ばれ、実測超過で捨てられ 1 試行を浪費する。マトリクス帯で一部救うが帯は初回超過後 1 回だけ。

6. **乗り遅れ再照会が保守的で良候補を刈る。**
   `_findRealBoarding` は同一路線・同一降車駅のみ対象、下流接続が崩れる多区間は候補ごと除外。安全側
   だが、徒歩最大の候補が落ちて速い経路へ寄る一因になる。

## ゼロから設計するなら

核心は「反応的に実測しながら推定を補正する」現行方式（#116/#117/#118 の積み重ね）を「先に測ってから
選ぶ」へ反転させること。

1. **目的関数を 1 つに固定。** 運動が主眼なら walk km（または kcal）最大化・到着 ≤ 締切を制約に。
   選定・タイブレーク・割増を全部この単位で揃える。
2. **「測ってから選ぶ」へ反転。** 直線距離で乗降候補駅を上位 K 個に絞り（フロンティア＝直線下限が
   予算内かつ ×1.4 上限が予算近傍）、origin→各乗車駅／各降車駅→goal の徒歩を最初に 1 回のマトリクスで
   一括実測。以降は実測値の上で決定的に選定する。迂回率学習・8 回ループ・割増ヒューリスティックが
   全て不要になり、バリアも最初から織り込まれ（限界5解消）、純粋関数でテストが楽、往復も最大 8 回→1 回。
   制約はマトリクス要素数課金（上限 25）なので「直線フロンティアで K を絞る」のが設計の肝。
3. **時間制約付き最長徒歩パスとして定式化。** 駅＝ノード、徒歩エッジ＝コスト小・歩行量大、乗車
   エッジ＝時間消費のグラフで「到着 ≤ 締切のもと徒歩量最大のラベル」を解く。複数路線を自然に同じ
   土俵へ乗せられる（限界2解消）。
4. **最小改善案。**（**#292 で実施済み**）いきなり再設計せずとも、`_baseForHybrid` を最速 1 本でなく
   上位数本へ広げるだけで限界2はかなり緩和する。→ #292 で「時間で上位N」ではなく**路線ファミリ別の
   代表 N 本**（`_basesForHybrid`）として実装した。素朴な上位N は同一ファミリの重複を掴むだけで多様性が
   増えない（#288）ため。

## 注意点

現行ロジックは #65/#67/#71/#115/#116/#117/#118 と、実データの汚さ（時刻欠落・運賃按分・乗り遅れ・
座標バリア）への実戦的バグ修正の集積。全面再設計はこれらが encode する現実のエッジケースへの回帰
リスクを伴う。推奨順序は (1) 限界1（目的関数 km/min）の判断と統一 → (2)「先に測ってから選ぶ」を
実測レイテンシ計測と併せて PoC。限界1は km か min かの設計判断が前提で、未決。
