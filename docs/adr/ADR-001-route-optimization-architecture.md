# ADR-001: ルート最適化アーキテクチャ比較・方式決定

- **ステータス:** 決定済み
- **日付:** 2026-05-16
- **関連 Issue:** #6（調査）, #7（RouteService定義）, #8（最適化ロジック実装）

---

## 背景

「時間内で最大限歩く」ルート最適化の実装方式が未定。以下2方式を比較し、#7〜#10 の実装前提として方式を確定する。

- **方式A:** クライアント完結（Google Directions API 直叩き + 端末側アルゴリズム）
- **方式B:** バックエンド構築（APIサーバーを介して最適化結果を返す）

---

## 評価軸と比較

| 評価軸 | 方式A: クライアント完結 | 方式B: バックエンド構築 |
|---|---|---|
| **APIキー秘匿** | × APKに埋め込み。パッケージID制限（Android: パッケージ名+SHA-1 / iOS: バンドルID）は不正利用をある程度抑制するが、APKデコンパイルでキー抽出は可能。HTTP リファラ制限はモバイルには使えない（Web専用）。**ただし本プロジェクトはすでに `google_maps_flutter` の APIキーを `AndroidManifest.xml` / `Info.plist` に埋め込んでいるため、Directions API 追加でセキュリティ上の立ち位置は変わらない** | ◎ キーはサーバー側にのみ存在 |
| **レート制限** | △ ユーザー端末から直接叩くため、ユーザー数増加時にクォータ超過リスクがある | ◎ サーバー側でキャッシュ・バッチ制御が可能 |
| **インフラコスト** | ◎ ゼロ（サーバー不要） | △ サーバー/コンテナ維持費が発生（月数百〜数千円〜） |
| **計算負荷** | ◎ 最適化ロジックは徒歩/電車数候補の比較のみで、端末負荷は軽微 | ◎ サーバー側で処理するため端末負荷なし（差異は実質ゼロ） |
| **実装コスト** | ◎ Flutter のみで完結、既存スタックで実装可能 | △ サーバー開発・デプロイ・認証が追加で必要 |
| **将来の #19（クラウド同期）との整合** | ◎ 履歴/設定/ストリークの同期は Firebase Auth + Firestore で独立して追加可能。最適化レイヤーと結合しない | △ 同期用バックエンドと最適化バックエンドを統合設計する場合はシナジーあり。ただし密結合になるリスクも |
| **テスタビリティ** | ◎ `RouteService` 抽象化（#7）でモックに差し替え可能。ビジネスロジックを Dart で完全にユニットテスト可能 | ○ サーバー側テストも必要になり総テスト量が増加 |
| **オフライン耐性** | △ API 呼び出しが必要（キャッシュは別途実装可能） | △ 同様に呼び出しが必要 |

---

## 最適化アルゴリズムの実現可能性（方式A）

「時間内で最大限歩く」は以下のフローで端末側のみで実現可能：

1. **徒歩ルートを取得**（Directions API `mode=walking`）→ 総所要時間・徒歩距離を確認
2. **予算内か判定**
   - 予算内 → 徒歩オンリーで walkRatio = 1.0 として採用
   - 予算超過 → 電車区間の候補を Directions API `mode=transit` で取得
3. **ハイブリッドルート最適化**
   - 電車区間（出発地→最寄り駅 / 乗換駅→目的地付近）を組み合わせて候補生成
   - 候補ごとに `totalMin ≤ budgetMin` を満たしつつ `walkKm` が最大となるものを選択
4. **指標算出**（walkRatio, kcal, segments, timelineNodes）を Dart で計算

API コール数は1ルート探索あたり最大2〜5回程度。Google Maps Platform の無料枠（$200/月相当）で個人利用・プロトタイプ規模であれば十分に収まる。

---

## 決定

**方式A（クライアント完結）を採用する。**

### 理由

1. **MVP フェーズのコスト最小化:** インフラ維持費ゼロで、開発リソースを UI/UX とロジックに集中できる
2. **計算量が端末で十分:** 最適化ロジックは候補数件の比較であり、端末負荷は問題にならない
3. **APIキーリスクは既に許容済み:** `google_maps_flutter` の時点でAPIキーをアプリに含めることを選択しており、Directions API追加でリスクは変わらない。パッケージID制限＋請求アラート＋クォータ上限設定で漏用時の被害を最小化する（HTTP リファラ制限はモバイル非対応のため対象外）
4. **#19との整合:** クラウド同期は Firebase で独立して実装でき、最適化レイヤーと結合させる必要がない
5. **#7のアーキテクチャ設計と一致:** `RouteService` を抽象化（#7）しておけば、将来バックエンドへの移行も可能

### APIキーの実運用上の緩和策

バックエンドを構築しない代わりに、以下を Google Cloud Console で設定する：

- **請求アラート:** 想定外のAPI費用をメールで即検知
- **クォータ上限:** 1日あたりの最大リクエスト数を設定し、漏洩時の被害を上限に抑える
- **パッケージID制限:** 正規アプリ以外からのリクエストを弾く（完全ではないが有効な抑止）

### 将来の移行パス

ユーザー数増加・セキュリティ要件の高まりなどでバックエンドが必要になった場合、`RouteService` 実装（#7）を差し替えるだけで段階移行できる。
